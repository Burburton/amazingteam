## 1. Architecture Review

### 1.1 What Works Well

| Aspect | Comment |
|--------|---------|
| Separation of concerns | User config vs foundation files is clean |
| Version locking | Explicit version in workflow file is simple and effective |
| Multiple access modes | CI (remote) + Local (downloaded) covers all scenarios |
| Merge approach | Deep merge for objects, replace for arrays is sensible |

### 1.2 Issues Found

#### Issue 1: Missing `opencode.jsonc` Generation

**Problem**: User project needs `opencode.jsonc` for OpenCode to work, but design doesn't specify how it's generated.

**Current Design**:
```
User Project:
├── .github/workflows/amazing-team.yml
├── amazing-team.config.yaml
└── src/
```

**Missing**: `opencode.jsonc` - OpenCode requires this file to know where to find instructions and skills.

**Solution Options**:
- A: Generate `opencode.jsonc` from `amazing-team.config.yaml` during `amazing-team init`
- B: Download foundation's `opencode.template.jsonc` and merge with user config
- C: Action generates it at runtime

**Recommendation**: Option A + C
- CLI generates `opencode.jsonc` during `init` for local dev
- Action generates it at runtime for CI

#### Issue 2: Skills Path Resolution

**Problem**: Skills in foundation reference local paths (`.opencode/skills/...`). When foundation is downloaded to different location, paths break.

**Example**:
```yaml
# In foundation's opencode.template.jsonc
skills:
  test-first-feature-dev:
    path: ".opencode/skills/test-first-feature-dev/SKILL.md"  # Relative path
```

**Solution**: Use absolute paths or runtime path resolution:
```javascript
// Action resolves paths at runtime
const skillPath = path.join(foundationDir, skill.relativePath);
```

#### Issue 3: Memory Directory Not Created

**Problem**: Foundation expects `.amazing-team/memory/{role}/` directories to exist, but they won't exist in user project.

**Solution**: Action must create memory directories during setup:
```javascript
const roles = ['planner', 'architect', 'developer', 'qa', 'reviewer', 'triage', 'ci-analyst'];
roles.forEach(role => {
  fs.mkdirSync(path.join(projectDir, '.amazing-team/memory', role), { recursive: true });
});
```

| Path resolution | Medium | Medium |

### 2.2 Technical Challenges

#### Challenge 1: Cross-Platform Path Handling

**Issue**: Windows uses `\`, Unix uses `/`. Foundation paths must work on both.

**Solution**: Use `path.join()` everywhere, never hardcode separators.

#### Challenge 2: Large Download Size

**Issue**: Foundation includes many files. Download time may affect CI speed.

**Solution**:
- Use GitHub Action caching
- Download only once per workflow run
- Consider splitting into core + overlays

#### Challenge 3: Offline Development

**Issue**: `amazing-team local` requires network. What if user has no network?

**Solution**:
- Cache downloads locally (`~/.amazing-team-cache/`)
- Allow specifying local foundation path: `amazing-team local --from /path/to/foundation`

## 4. Risk Assessment Review

### 4.1 Additional Risks Not Covered

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| NPM registry downtime | CI fails | Low | Fallback to GitHub releases |
| Breaking change in OpenCode | Action fails | Low | Version pin OpenCode action |
| User deletes `opencode.jsonc` | Local dev breaks | Medium | `amazing-team validate` catches this |
| Git conflicts in workflow file | Upgrade fails | Medium | CLI handles conflicts gracefully |
| Foundation security vulnerability | All projects affected | Low | Automated security scanning |

### 4.2 Mitigation Strategies

```
Risk Mitigation Flow:
    │
    ▼
1. Pre-flight checks (validate config, check network)
    │
    ▼
2. Download with retry (3 attempts, exponential backoff)
    │
    ▼
3. Verify download (checksum validation)
    │
    ▼
4. Merge configs (validate result)
    │
    ▼
5. Create runtime directories
    │
    ▼
6. Generate opencode.jsonc
    │
    ▼
7. Run OpenCode
```

## 5. Open Questions Resolution

### Question 1: Action Hosting

**Options**:
- A: Separate repo
- B: Included in foundation

**Analysis**:
| Factor | Separate Repo | Included |
|--------|---------------|----------|
| Versioning | Independent | Tied to foundation |
| Development | Separate workflow | Simpler |
| Release | More complex | Single release |
| User experience | Two references | One reference |

**Decision**: Option B (Included in foundation)

**Rationale**: Simpler for users, single source of truth, easier versioning.

### Question 2: Version Source

**Options**:
- A: NPM registry
- B: GitHub releases

**Analysis**:
| Factor | NPM | GitHub |
|--------|-----|--------|
| Reliability | High | High |
| Speed | Fast | Fast |
| Auth required | No | No |
| Caching | Built-in | Manual |

**Decision**: Support both, prefer NPM

**Rationale**: NPM is standard for JS packages. GitHub as fallback for edge cases.

### Question 3: Overlay Handling

**Options**:
- A: Specify in action config
- B: Include in amazing-team.config.yaml

**Decision**: Option B (Include in config)

**Rationale**: Consistent with other settings, version controlled with project.
